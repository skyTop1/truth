# uni-app 项目技术选型与 UI 框架分析

版本：v0.1  
更新时间：2026-04-14

## 1. 文档目标

本文档用于确定这个新项目的第一版技术路线，重点回答两个问题：

1. 新项目应该使用哪一条 uni-app 技术主线
2. UI 框架应该选哪一个，为什么

这份文档会作为后续讨论的基础版本，后面可以继续按你的业务类型、目标平台、设计风格、团队习惯逐步修改。

配套重规划文档：

- `方案 A` 的进一步落地规划与主题色体系见 `docs/方案A-项目重规划与主题色体系.md`

## 2. 当前结论

当前更推荐的主线是：

- `uni-app` 标准版
- `Vue 3 + TypeScript + Vite`
- `Composition API + <script setup>`
- `Pinia`
- `uni-ui` 作为基础 UI 框架
- 在 `uni-ui` 之上再封装一层业务组件，不建议一开始就把项目完全绑死在第三方整套 UI 风格上

一句话结论：

> 如果这是一个要长期维护、覆盖小程序/H5/App 的正式项目，第一版建议优先走 `uni-app Vue3/Vite + Pinia + uni-ui`，而不是直接 all-in `uni-app x` 或直接重度绑定第三方成套 UI 框架。

## 3. 为什么不是默认上 uni-app x

`uni-app x` 确实代表更前沿的方向，核心优势是：

- `uts + uvue` 体系更接近原生能力
- 官方明确定位为“下一代 uni-app”
- 在 App 端强调原生渲染和高性能

但它当前不适合作为本项目第一版默认主线，原因也很明确：

- 官方文档写明：`uni-app x` 在 App 平台“暂不支持 `pinia` 和 `vuex`”
- `uni-app x` 的页面体系是 `.uvue` / `main.uts`，工程和生态与标准 uni-app 已经不是完全同一套
- 现阶段成熟的 UI 框架、社区案例、历史沉淀仍然更多集中在标准 `uni-app Vue3` 路线

因此当前建议是：

- 第一版先用成熟路线快速落地业务
- 如果后续你明确是“App-first、高性能原生体验优先、并且愿意接受生态收缩”，再单独评估 `uni-app x`

## 4. 推荐技术栈

### 4.1 主技术路线

推荐：

- 框架：`uni-app` 标准版
- 语法：`Vue 3`
- 工程化：`Vite`
- 语言：`TypeScript`
- 组件写法：`Composition API + <script setup>`
- 状态管理：`Pinia`

原因：

- 官方对 `Vue3/Vite` 已有明确支持
- 官方对 `Pinia` 有明确文档支持
- 这是现在 uni-app 新项目里最合理、最现代、也最容易持续演进的一条路线

### 4.2 工程建议

建议同时保留两种开发方式的兼容性：

- 日常工程开发以 `CLI/Vite` 为主
- 真机调试、发版、平台特性处理时保留 `HBuilderX` 工作流

原因：

- `CLI/Vite` 更适合现代工程化
- `HBuilderX` 在 uni-app 生态里仍然是重要工具链

### 4.3 状态管理建议

推荐：

- 全局状态：`Pinia`
- 页面级复杂逻辑：抽离为 `composables`
- API 请求层：自己封装统一请求模块，不要把网络请求散落在页面里

不建议：

- 新项目继续用 `Vuex`
- 所有状态都堆到一个大 store

## 5. UI 框架选型结论

### 5.1 最终推荐

首选方案：

- `uni-ui` 作为基础组件层

备选方案：

- `uview-plus` 适合想要更多现成组件和业务组件的团队
- `FirstUI` 适合交付导向、接受商业版授权、希望快速拿到较完整中后台/业务模板的团队
- `ThorUI` 适合已有使用经验的团队，但不建议作为当前新项目的默认首选

### 5.2 我给你的建议

如果你现在没有强历史包袱，我建议直接这么定：

- 底层 UI：`uni-ui`
- 视觉规范：自己定义品牌色、圆角、阴影、间距、字体层级
- 业务组件：自己封装 `AppHeader`、`AppCard`、`AppFormItem`、`AppEmpty`、`AppList`、`AppFilterBar` 等

这样做的好处是：

- 不会被第三方框架默认视觉风格锁死
- 后期换皮、品牌升级、B 端/C 端切换成本更低
- 复杂业务场景里，不会出现大量覆盖第三方样式的维护负担

## 6. UI 框架对比

下面的评分是基于官方资料和工程经验做出的工程判断，不是官方评分。

| 方案 | 定位 | 优势 | 风险/限制 | 适合场景 | 结论 |
| --- | --- | --- | --- | --- | --- |
| `uni-ui` | DCloud 官方扩展组件库 | 官方体系、跨端一致性强、性能导向、和 uni-app 生态耦合最顺 | 视觉偏基础，成套业务组件不如第三方丰富 | 长期维护型项目、跨端正式项目 | `最推荐` |
| `uview-plus` | 社区维护的 uView Vue3/plus 路线 | 组件丰富、上手快、业务场景覆盖广、文档相对完整 | 官方仓库明确说明“时间有限”，维护强依赖社区；4.x 升级路线存在持续演进成本 | 希望快速堆业务页面、团队已有 uView 经验 | `可选，但不是第一推荐` |
| `FirstUI` | 跨平台组件库，商业版能力较强 | 组件和布局较多、主题能力较好、交付效率高 | 大量能力区分开源版/VIP 版，长期授权和团队协作成本需要提前确认 | 商业项目、模板驱动开发、交付优先 | `预算充足时可选` |
| `ThorUI` | 轻量型组件库 + 模板生态 | 组件不少，模板和示例丰富，商用规则相对清晰 | 开源/会员边界明显，部分能力要会员；官方文档公开页更新时间偏早 | 团队已有 ThorUI 使用经验 | `不建议作为当前默认首选` |

## 7. 各 UI 框架分析

### 7.1 uni-ui

推荐级别：`5/5`

优点：

- DCloud 官方出品，和 uni-app 官方生态配套最完整
- 官方明确强调性能、全端、自适应和 DCloud 规范一致性
- 官方还提供 `uni ui项目模板`，说明它本身就是官方推荐的日常开发路线之一
- 更适合作为“基础层”，避免项目早期样式体系被第三方框架完全主导

缺点：

- 视觉表现更偏“基础设施型”
- 如果你要做非常丰富的业务组件，通常仍然需要自己再封装

适合：

- 新项目
- 中长期维护
- 需要跨端稳定性
- 希望保留设计系统主导权

### 7.2 uview-plus

推荐级别：`3.5/5`

优点：

- 组件覆盖面广
- 文档体验不错
- 对很多常见业务页面更友好
- 支持选项式和组合式写法，迁移成本相对可控

缺点：

- 它本质是 `uView2.0` 的 plus 路线
- 官方 GitHub README 明确写了“时间有限”“可能一个月定期维护几次”
- 文档中 4.x 仍写着“预计 2025 年 08 月发布”，说明路线推进存在不确定性

适合：

- 团队之前做过 uView 项目
- 需要大量现成组件快速出页面
- 接受社区维护节奏

不适合：

- 对长期维护稳定性要求很高
- 不希望把 UI 基础设施建立在社区 fork 路线上

### 7.3 FirstUI

推荐级别：`3.5/5`

优点：

- 组件非常多
- 主题和布局能力较完整
- 文档中明确写了支持 `vue2` / `vue3`，并支持 `vue3` 组合式 API
- 对追求交付效率的团队很友好

缺点：

- 官方文档明确区分开源版和商业版
- 不少高级组件是 `VIP专属`
- 如果项目后续多人协作、外包交付、源码流转较多，需要提前把授权边界确认清楚

适合：

- 已确认采购商业授权
- 希望更多现成模板和业务组件
- 以交付效率为第一目标

### 7.4 ThorUI

推荐级别：`3/5`

优点：

- 组件数不少
- 对常见移动端页面场景支持不错
- 模板生态比较实用

缺点：

- 官方文档中明确区分开源组件和会员组件
- 文档公开页可见更新时间较早
- 对新项目来说，默认选择它的收益不如 `uni-ui` 或 `FirstUI`

适合：

- 团队已熟悉 ThorUI
- 现成模板正好符合业务

## 8. 推荐落地方案

### 方案 A：最稳妥，也是我当前最推荐的方案

- 主技术：`uni-app + Vue3 + TS + Vite`
- 状态：`Pinia`
- UI：`uni-ui`
- 策略：`官方基础组件 + 自定义业务组件层`

适合：

- 企业项目
- 电商/内容/工具类 App
- 同时要做小程序、H5、App
- 希望后续逐步扩展设计系统

### 方案 B：快速交付优先

- 主技术：`uni-app + Vue3 + TS + Vite`
- 状态：`Pinia`
- UI：`FirstUI` 或 `uview-plus`

适合：

- 需求明确
- 页面数量多
- 上线时间紧
- 团队愿意接受第三方风格和依赖

### 方案 C：原生性能优先

- 主技术：`uni-app x + uts + uvue`
- 状态：按官方建议使用 `globalData` 或专用模块
- UI：优先自己封装，谨慎选择第三方

适合：

- App-first
- 原生能力和性能优先级极高
- 可以接受生态和开发范式变化

不建议作为当前默认第一版方案。

## 9. 建议的项目结构

如果按方案 A 落地，建议从第一天就按下面的结构组织：

```text
├─ src 或项目根目录
│  ├─ api                # 接口定义与请求封装
│  ├─ components         # 通用业务组件
│  ├─ composables        # 组合式逻辑
│  ├─ constants          # 常量
│  ├─ enums              # 枚举
│  ├─ hooks              # 平台/页面钩子封装
│  ├─ pages              # 页面
│  ├─ pages-sub          # 分包页面
│  ├─ static             # 静态资源
│  ├─ stores             # Pinia
│  ├─ styles             # 主题、变量、原子类
│  ├─ types              # TS 类型
│  ├─ utils              # 工具函数
│  ├─ App.vue
│  ├─ main.ts
│  ├─ manifest.json
│  ├─ pages.json
│  └─ uni.scss
├─ docs
└─ package.json
```

## 10. 设计系统建议

不管最后选哪个 UI 框架，我都建议把下面这些设计令牌先独立出来：

- 主色、成功色、警告色、危险色
- 页面背景色、卡片背景色、边框色
- 字号层级
- 圆角体系
- 间距体系
- 阴影体系
- 暗黑模式变量

原因：

- 这样做可以把“框架能力”和“品牌视觉”分开
- 后面如果你要换 UI 框架，不至于整站重写

## 11. 当前建议的决策顺序

建议你下一步先确认这 4 件事：

1. 目标平台是否必须同时覆盖 `微信小程序 + H5 + App`
2. 第一版是更偏 `快速上线`，还是更偏 `长期维护`
3. 是否接受 `商业版/VIP` UI 框架授权
4. 你的项目偏 `C端产品`、`B端系统`、`商城`、`内容社区` 还是 `工具类应用`

如果你现在不想细分，我建议先默认这样定：

- 技术主线：`uni-app Vue3/Vite`
- 状态管理：`Pinia`
- UI 框架：`uni-ui`
- 项目策略：`基础组件官方化，业务组件自定义化`
