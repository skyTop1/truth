# 赛博祭祖 App / 小程序产品概念与体验设计

> 状态说明：本文件已归档，不再作为当前主线产品方案。当前项目以“热点型、赛博祭祖、本地优先、不上传用户内容”的方向为准。

版本：v0.1  
更新时间：2026-04-14

## 1. 文档目标

本文档用于把“赛博祭祖”这个方向，从一句概念，推进到可执行的产品与体验设计方案。

本文档不解决技术选型问题，重点解决以下问题：

1. 这个产品应该服务谁
2. 这个产品的核心体验应该是什么
3. 小程序和 App 各自适合承接哪些能力
4. 首页、祭拜、纪念馆、家谱、提醒等模块应该如何组织
5. 视觉风格怎样做到“有赛博感，但不轻浮”

一句话目标：

> 把“纪念、祭拜、传承、家族连接”做成一个庄重、克制、具有未来感的数字祠堂，而不是做成猎奇的赛博玩具。

---

## 2. 产品定义

### 2.1 一句话定位

这是一个面向华人家庭的数字纪念与祭祖产品，提供：

- 数字祠堂
- 周年祭日提醒
- 线上祭拜流程
- 家族记忆档案
- 家谱关系整理
- 亲友协同纪念

### 2.2 对外品牌建议

“赛博祭祖”适合作为内部概念，不建议直接作为面向用户的正式品牌名。

建议对外品牌更克制、更有文化感，例如：

- `云祠`
- `灯序`
- `归祠`
- `念堂`
- `祠光`

建议策略：

- 内部代号保留 `赛博祭祖`
- 对外品牌采用更稳重的名称
- 副标题使用“数字纪念空间”“线上家祠”“家族记忆馆”这一类表达

### 2.3 产品关键词

- 庄重
- 纪念
- 传承
- 陪伴
- 家族连接
- 克制的未来感

## 3. 用户与核心场景

### 3.1 核心用户

#### 用户 A：家族组织者

典型角色：

- 家中长辈
- 家族微信群的组织者
- 负责节日祭拜和纪念日提醒的人

核心诉求：

- 想把祭日、忌日、周年、家谱节点整理清楚
- 想让家族成员在不同城市也能参与
- 想留住先人的照片、故事、音频和家族记录

#### 用户 B：异地工作或留学的年轻一代

典型角色：

- 常年不在老家
- 对传统祭拜有情感，但缺少参与条件
- 希望有更轻量、更现代的表达方式

核心诉求：

- 想在节日或纪念日表达心意
- 想保留和查看家族记忆
- 希望流程简单，不繁琐

#### 用户 C：家庭纪念空间维护者

典型角色：

- 负责上传资料的人
- 家族史、照片、旧信件、录音的整理者

核心诉求：

- 想有结构化的存档空间
- 想让内容安全、可控、可分享
- 想管理谁能看、谁能编辑、谁能留言

### 3.2 核心使用场景

- 清明、冬至、中元、春节等时间点的线上祭拜
- 某位亲人的周年纪念日提醒
- 家族成员异地共同上香、献花、留言
- 家谱关系梳理和先人生平整理
- 长辈照片、录音、故事的长期留存

### 3.3 不做的方向

- 不做重娱乐化“养成式”祭拜
- 不做夸张的灵异视觉和惊悚风格
- 不做过强游戏化积分体系
- 不做低俗分享和公开围观型产品

## 4. 产品原则

### 4.1 体验原则

- 所有祭拜动作都要短而有仪式感
- 所有页面都要优先尊重情绪，不刺激、不吵闹
- 所有动效都要慢、轻、稳，不能像电商活动页
- 所有记录都要支持“家族私密空间”优先

### 4.2 业务原则

- 先做数字纪念空间，再扩展为家族记忆系统
- 先保证小程序轻量可达，再在 App 承担深度内容
- 先做好提醒、纪念、家族协同，再考虑 AI 衍生能力

### 4.3 伦理原则

- 默认尊重逝者与家庭隐私
- 默认关闭公开发现页
- 默认以家庭邀请制为核心
- 如果后续引入 AI 生平整理、音频转写、纪念文本生成，必须明确标注“AI 辅助生成，需人工确认”

## 5. 平台策略

### 5.1 小程序定位

小程序负责：

- 节点提醒
- 快速祭拜
- 家族祠堂浏览
- 轻量留言
- 家族成员邀请
- 纪念馆分享

小程序特点：

- 轻启动
- 易分享
- 适合节日高频触达
- 适合拉家族成员一起参与

### 5.2 App 定位

App 负责：

- 深度档案整理
- 家谱编辑
- 照片、录音、视频资料管理
- 长文生平整理
- 通知中心和更完整提醒设置
- 个性化纪念空间配置

App 特点：

- 更适合沉浸式浏览
- 更适合大体量内容沉淀
- 更适合长期留存

### 5.3 结论

推荐产品结构：

- `小程序 = 入口 + 节日使用 + 家族协同`
- `App = 深度整理 + 沉淀 + 长期留存`

## 6. UI Direction Summary

### 6.1 气质方向

不做“霓虹夜店风赛博”。

做“未来祠堂感”：

- 深色夜幕
- 微金色烛火
- 青蓝色电子微光
- 像数字碑刻、电子祠堂、云端灯火

关键词：

- 静谧
- 深远
- 庄重
- 未来感
- 微光
- 记忆感

### 6.2 视觉概念

推荐主题名：

`夜祠金`

推荐视觉意象：

- 暗色空间中的一盏灯
- 层叠的电子牌位
- 微弱流动的香火光迹
- 像数据流一样延展的家谱连线
- 像旧照片和电子档案混合的纪念卡片

### 6.3 风格边界

应该有：

- 细腻的暗色层次
- 克制的金色强调
- 少量青蓝冷光用于“数字感”
- 轻量的烟雾/微粒/波纹动画

不应该有：

- 大面积荧光紫
- 高饱和电音风渐变
- 夸张闪烁和强动效
- 游戏化 UI 外观

## 7. Layout And Responsive Plan

### 7.1 信息架构

建议一级结构：

1. `家祠`
2. `祭拜`
3. `家谱`
4. `纪念馆`
5. `我的`

### 7.2 首页：家祠

首页不是资讯流，而是“今日纪念入口”。

顶部结构：

- 家族名称 / 数字祠堂名
- 今日日期与农历信息
- 今日待纪念事件

核心区域：

- 当前主祭对象卡片
- 快捷祭拜按钮
- 本月纪念日时间轴
- 家族动态摘要

底部区域：

- 亲友最近留言
- 家族记忆精选

### 7.3 祭拜页

祭拜页强调流程感，建议拆成三段：

1. 选择对象
2. 选择仪式动作
3. 留言与完成反馈

推荐动作：

- 上香
- 献花
- 点灯
- 敬茶 / 敬酒
- 鞠躬
- 默哀
- 写一句心里话

完成后反馈：

- 今日祭拜已完成
- 已同步到家族纪念动态
- 可选择分享给家人共同参与

### 7.4 家谱页

家谱页不做复杂表格，优先做“关系图 + 人物卡片”。

交互结构：

- 默认展示核心家族树
- 点击人物节点展开人物卡
- 可切换“谱系模式 / 纪念模式”

### 7.5 纪念馆页

纪念馆像一个数字展陈空间，承接长期资料。

模块建议：

- 生平简介
- 时间轴
- 照片墙
- 语音 / 视频
- 家人留言
- 重要物件或故事

### 7.6 我的页

承接配置与协同：

- 我的家族
- 提醒设置
- 受邀空间
- 隐私权限
- 资料导入
- 纪念内容管理

### 7.7 响应式策略

小程序 / 手机 App：

- 单列布局
- 核心 CTA 固定底部
- 卡片宽度优先撑满

平板 / 大屏 H5：

- 首页与纪念馆可升级成双列布局
- 左侧信息总览，右侧详情内容

## 8. Component Behavior Spec

### 8.1 `AppMemorialHero`

用途：

- 首页主纪念对象入口

内容：

- 人物名
- 称谓
- 生卒信息
- 今日状态标签
- 快捷祭拜按钮

状态：

- 默认
- 今日纪念日
- 已完成祭拜
- 无资料

### 8.2 `AppRitualActionCard`

用途：

- 祭拜动作卡片

动作类型：

- 上香
- 献花
- 点灯
- 敬茶
- 留言

交互规则：

- 点击后出现轻反馈动画
- 不做夸张音效和震动
- 完成后卡片转入“已完成”态

### 8.3 `AppAncestorCard`

用途：

- 家谱节点人物卡

内容：

- 头像 / 牌位图形
- 姓名
- 称谓
- 生卒年份
- 与当前用户关系

状态：

- 常规
- 当前查看中
- 已故纪念重点
- 内容未完善

### 8.4 `AppMemoryTimeline`

用途：

- 纪念馆时间轴

内容：

- 时间
- 标题
- 照片 / 音频 / 文本
- 编辑者

交互规则：

- 默认按时间排序
- 支持按“生平 / 家族记忆 / 节点纪念”切换筛选

### 8.5 `AppFamilyNotice`

用途：

- 首页提醒与家族动态摘要

内容：

- 即将到来的祭日
- 家族新增留言
- 家族资料待完善提醒

### 8.6 `AppPrivacyBadge`

用途：

- 强化“私密 / 家族可见 / 邀请可见”的边界

要求：

- 所有涉及照片、语音、生平内容的页面，都要能看见权限标识

## 10. Accessibility And UX Checks

### 10.1 需要重点保证

- 文字对比度稳定，暗色背景不能糊成一片
- 长辈用户可用性优先，点击区域要大
- 祭拜流程不能超过 3 步核心动作
- 重要按钮文案必须直接，不要写成抽象词

### 10.2 必须覆盖的状态

- 空状态
- 无家族空间状态
- 无纪念对象状态
- 今日无祭日状态
- loading
- 提交成功
- 提交失败
- 权限不足

### 10.3 交互节奏

- 首页强调“安静浏览 + 快捷进入”
- 祭拜页强调“连续动作 + 完成反馈”
- 纪念馆强调“沉浸阅读 + 资料浏览”

### 10.4 可访问性细节

- 图标必须有语义标签或明确装饰属性
- 关键按钮颜色不能单独承担含义，必须配文案
- 留言、上传、邀请等流程必须有明确成功反馈

## 11. MVP 范围建议

### 11.1 第一阶段必须做

- 登录与家庭空间加入
- 首页家祠总览
- 祭日提醒
- 一键祭拜流程
- 纪念对象卡片
- 家族留言
- 基础纪念馆

### 11.2 第二阶段再做

- 家谱关系图
- 照片 / 音频 / 视频档案
- 家族多人协同编辑
- 更完整的提醒中心

### 11.3 第三阶段谨慎评估

- AI 生成纪念文案
- AI 整理生平摘要
- 旧照片修复
- 语音转文字

要求：

- 所有 AI 功能必须后置
- 必须是辅助整理，不得替代真实家庭记录

## 12. Implementation Plan

### 12.1 页面落地优先级

1. `家祠首页`
2. `祭拜流程页`
3. `纪念馆详情页`
4. `我的 / 提醒设置`
5. `家谱页`

### 12.2 组件优先级

优先新增：

- `AppIcon`
- `AppMemorialHero`
- `AppRitualActionCard`
- `AppAncestorCard`
- `AppMemoryTimeline`
- `AppFamilyNotice`
- `AppPrivacyBadge`

复用现有：

- `AppButton`
- `AppCard`
- `AppDialog`
- `AppEmpty`
- `AppNavBar`
- `AppTabBar`
- `AppTag`

### 12.3 主题落地建议

当前项目已有一套偏沉稳蓝灰的主题基础。

如果确认转向本方案，下一步建议：

1. 新建“赛博纪念”主题 token 草案
2. 先替换首页和祭拜页的视觉变量
3. 保留现有组件结构，只重做 token 和局部组件样式
4. 先让首页、祭拜页、纪念馆形成统一风格，再扩展家谱页

### 12.4 设计风险

- 如果赛博感过强，会破坏庄重气质
- 如果传统感过强，会失去年轻用户接受度
- 如果仪式流程太长，小程序转化会很差
- 如果公开分享太强，会触碰隐私和伦理边界

---

## 13. 最终建议

这个项目最关键的不是“把祭拜做线上”，而是把“家族纪念关系”做成一个可长期存在的数字空间。

正确方向应该是：

- 用小程序承接节日与协同
- 用 App 承接沉淀与档案
- 用克制的未来感替代廉价赛博风
- 用真实家族记录替代表演化仪式

一句话结论：

> 这不是一个“线上烧香工具”，而应该是一个“家族记忆、纪念与连接的数字祠堂”。
