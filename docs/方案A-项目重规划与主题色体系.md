> 状态说明：本文件保留为历史 starter 阶段主题规划归档。当前项目主视觉已切换到赛博祭祖方向，实际实现请以当前代码和赛博祭祖文档为准。

## 1. 文档目标

本文档基于上一版选型文档中的 `方案 A` 进行进一步规划，目标是把项目从“技术选型阶段”推进到“可执行的落地规划阶段”。

本次规划的重点不是页面功能，而是：

- 明确 `方案 A` 的工程边界
- 先把主题颜色体系设计好
- 为后续页面、组件、品牌升级留出统一扩展口

一句话目标：

> 先把颜色系统定成“可维护的设计资产”，再去做页面，而不是先做页面再回头修主题。

## 2. 方案 A 确认

本项目当前默认采用：

- 框架：`uni-app` 标准版
- 语法：`Vue 3`
- 语言：`TypeScript`
- 工程化：`Vite`
- 状态管理：`Pinia`
- 基础组件：`uni-ui`
- UI 策略：`官方基础组件 + 自定义业务组件层`

本方案的核心思想不是“选一个大而全 UI 库直接套完”，而是：

- 让 `uni-ui` 负责跨端基础能力
- 让主题色系统负责统一视觉
- 让业务组件负责页面风格一致性

## 3. 重规划原则

### 3.1 项目原则

- 移动端优先，兼顾 H5 / 小程序 / App
- 第一版优先保证长期维护，而不是只追求短期出图速度
- 避免重度绑定第三方默认样式
- 所有颜色必须语义化，不能在页面里散落硬编码色值

### 3.2 主题原则

- 主题色需要“可信、现代、耐看”，不能过于花哨
- 色彩系统需要支持后续品牌升级
- 颜色不仅服务视觉，还要服务状态表达和交互反馈
- 颜色方案要优先保证可读性和跨端稳定表现

### 3.3 当前默认前提

由于你还没有明确业务类别和品牌方向，这一版先采用一个更通用的默认基调：

- 风格方向：`理性、可信、沉稳、克制`
- 主题倾向：`黛蓝为主，灰青为辅`
- 页面气质：`浅灰底、低饱和、秩序感强、信息层级清晰`

这个方向适合大多数正式项目，包括：

- 工具类应用
- 企业服务
- 内容平台
- 商城与会员体系

如果后续你告诉我项目更偏：

- 医疗
- 金融
- 社区
- 国风
- 潮流电商

我再按行业特征重调色板。

## 4. 主题颜色总策略

本项目建议采用“三层颜色结构”：

1. `品牌层`
2. `语义层`
3. `界面层`

### 4.1 品牌层

品牌层只负责识别感，不直接承担所有页面颜色。

建议：

- 主品牌色：偏沉稳的低饱和黛蓝
- 辅助品牌色：偏克制的灰青色
- 强调色：仅用于少量高价值操作和数据高亮

### 4.2 语义层

语义层用于表达交互和状态：

- `primary`
- `success`
- `warning`
- `danger`
- `info`

要求：

- 颜色一定要和行为含义绑定
- 同一种语义在全项目中不能变来变去

### 4.3 界面层

界面层决定页面观感，主要包括：

- 页面背景
- 卡片背景
- 分割线
- 一级文字
- 二级文字
- 禁用态
- 浮层和遮罩

界面层必须克制，不能过度依赖彩色背景，否则跨端后期会很难维护。

## 5. 推荐主题方向

### 5.1 推荐主方案

当前推荐主题名：

`黛蓝灰`

这是一个偏长期主义的配色方向，核心感受是：

- 黛蓝建立信任感、专业感和稳定感
- 灰青负责轻微的现代感和呼吸感
- 中性色保持信息密度和可读性

它的优势是：

- 比亮蓝更克制
- 比高饱和青绿更稳重
- 比冷黑灰更有品牌识别
- 在小程序和 App 中都比较稳

### 5.2 颜色角色分工

- `黛蓝`：主按钮、主操作、链接、选中态、品牌识别
- `灰青`：轻强调、统计高亮、图表辅助、局部渐变
- `中性色`：背景、卡片、边框、正文、弱化信息

## 10. 工程落地方式

### 10.1 文件规划

建议后续这样组织：

```text
styles/
├─ tokens.scss           # 原始设计令牌
├─ semantic.scss         # 语义变量
├─ theme-light.scss      # 浅色主题
├─ theme-dark.scss       # 暗色主题
└─ index.scss            # 统一导出
```

### 10.2 变量层级

建议使用三层变量：

1. 原始色值层  
   例如：`$brand-600`, `$gray-700`

2. 语义变量层  
   例如：`$color-primary`, `$color-text-secondary`

3. 组件变量层  
   例如：`$button-primary-bg`, `$card-border-color`

这样做的好处：

- 改品牌色不影响组件逻辑
- 改组件策略不影响基础色板
- 后续做暗黑模式更容易切换

### 10.3 落地约束

- 页面和组件内禁止直接写十六进制颜色
- 所有颜色必须走变量
- 状态色不能被业务页面随意重定义
- 渐变色只允许出现在品牌区、活动区、头图区

## 11. 第一阶段执行建议

如果下一步开始真正搭项目，建议按下面顺序推进：

1. 先创建主题变量和 `uni.scss` 桥接层
2. 先实现 `AppButton`、`AppCard`、`AppInput` 三个基础业务组件
3. 再实现导航、列表、表单、空状态
4. 最后再给具体页面铺主题

原因：

- 先定组件，后做页面，能减少返工
- 主题先固化，后面页面产出会快很多
