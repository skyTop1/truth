# 赛博祭祖本地优先产品缺口清单

版本：v0.1  
更新时间：2026-04-14

## 0. 最新落地进展

截至本轮迭代，以下能力已经落地，不再属于“完全未开始”状态：

- 已补齐统一本地存储层、local id 工具和祭祖归档 store
- 已定义祖先节点、祭祖记录、海报记录等核心本地数据结构
- 首页统计、本机动态、最近祖先节点已经改为读取本地记录
- 仪式页支持自定义祖先称呼、自由输入文案，并可复用本地历史模板
- 结果页生成后会把祭祖记录与海报状态写入本地祖域仓
- 祖域页已经从概念占位页升级为本地管理入口，可查看历史、查看海报、复用历史、删除单条记录、删除单张海报、清空全部记录
- 已增加“复制备份串”能力，用于手动导出当前本地 JSON 快照
- 已补齐本地归档版本迁移与损坏缓存自动恢复策略
- 结果页已补齐海报缓存恢复、缓存失效、生成失败、相册权限受限等关键状态
- 首页与祖域页已增加首次进入、已有历史但无海报、完整闭环等状态矩阵提示
- 祖域页已支持手动导出与导入恢复，本地备份链路形成闭环

但这不代表清单已经全部完成。

当前仍然需要继续推进的是：

- 更完整的状态矩阵覆盖
- 更明确的本地隐私说明和发布前权限说明
- README 的更完整使用说明与备份说明

补充说明：

- `src/api/*` 当前已明确标记为未启用占位，并在代码层默认禁用远程请求

## 1. 文档目标

本文档用于在当前方向已经明确的前提下，整理本项目下一阶段仍未完善的内容。

这份清单不再讨论“要不要做云端产品”或“要不要接接口”，而是基于当前已确认约束，明确：

1. 哪些内容已经确定，不再反复争论
2. 哪些内容当前不算缺口
3. 哪些内容必须继续补齐
4. 后续应该按什么顺序推进

一句话目标：

> 把 `Truth` 从“能跑的赛博祭祖原型”推进到“方向统一、体验完整、隐私叙事明确、可持续迭代”的本地优先小程序 / H5 产品。

---

## 2. 已确认前提

以下前提已经确认，后续规划默认全部遵守：

- 项目名称固定为 `Truth`
- 产品概念固定为“赛博祭祖”
- `Truth` 作为项目名，与“赛博祭祖”产品概念不冲突
- 当前阶段不接任何接口
- 当前阶段不引入账号体系
- 当前阶段不把用户祭祖内容上传到服务端
- 数据只保存在本地与本地缓存
- 不介入用户祭祖内容，不做服务端审核
- 用户想怎么祭祖就怎么祭祖
- 优先平台仍然是 `微信小程序 + H5`

---

## 3. 当前不算缺口的内容

以下内容不纳入当前阶段缺口：

- 云端接口接入
- 账号登录 / 注册
- 远程用户体系
- 服务端数据存储
- 内容上传到云端
- 远程内容审核
- 远程统计后台
- 多端云同步

这些方向不是“还没做”，而是当前阶段明确“不做”。

---

## 4. 缺口清单

## P0 必须先统一

### 4.1 规则文件还没写清“本地优先”

当前项目规则已经定义了技术栈、目录职责、图标来源和主题策略，但还没有把“禁止引入接口、禁止上传用户祭祖内容、默认本地缓存”写成硬约束。

需要补齐：

- 在 `AGENTS.md` 中增加“本地优先”规则
- 在 `shrimp-rules.md` 中增加同样约束
- 明确后续页面和功能默认以本地缓存为第一选择
- 明确后续如果要引入接口，必须先单独评审

### 4.2 文档方向仍有冲突

当前热点方向已经明确，但仓库里仍保留一份“庄重、长期、数字纪念空间”的旧文档，会持续干扰后续判断。

需要补齐：

- 归档或重写旧版产品概念文档
- 统一所有文档中的产品口径
- 明确当前主线是“热点型、本地优先、赛博祭祖体验”

### 4.3 README 和应用描述仍停在 starter 阶段

当前 `README.md` 和 `manifest.json` 仍然带有明显的骨架项目描述，没有把产品定位切到当前方向。

需要补齐：

- README 改成 `Truth` 当前真实定位
- README 明确说明“本地优先，不上传祭祖内容”
- `manifest.json` 里的描述改成真实产品说明
- 说明当前主要目标平台和当前阶段范围

### 4.4 主题叙事仍然双轨并存

实现层已经是赛博祭祖视觉方向，但文档里仍保留着一套“黛蓝灰、沉稳正式”的旧叙事。

需要补齐：

- 统一“当前主主题”与“历史主题方案”的关系
- 决定旧主题是保留为历史方案，还是从当前主文档中拆出
- 避免后续设计继续在“热点赛博”和“正式沉稳”之间来回摇摆

---

## P1 产品体验必须补齐

### 4.5 祖域第二入口仍是概念占位

当前第二个 tab 已经不叫“工作台”，方向是对的，但内容本质上仍是占位页。

需要补齐：

- 把祖域页做成真正的第二入口
- 承接本地祖先节点列表
- 承接本地祭祖历史
- 承接本地海报图库
- 承接常用模板或常用仪式动作

### 4.6 首页仍是静态展示页

当前首页视觉和气质已经立住，但多数数据还是写死内容。

需要补齐：

- 首页统计改成来自本地记录
- 最近祖域动态改成来自本地历史
- 根据本地是否已有记录展示不同首屏状态
- 支持“首次进入”和“已有祭祖记录”的差异化展示

### 4.7 仪式页还不够自由

当前仪式页已经可以完成一次体验，但更像固定流程演示，还不够符合“用户想怎么祭祖就怎么祭祖”的前提。

需要补齐：

- 支持自定义祖先称呼
- 支持自定义悼词模板或自定义短句
- 支持扩展仪式动作，而不是只保留固定三项
- 支持本地保存用户常用仪式组合

### 4.8 结果页还没形成完整本地闭环

结果页已经能生成海报、预览和保存，但这次结果还没有真正写回本地历史。

需要补齐：

- 生成结果后写入本地祭祖记录
- 保存海报后写入本地海报记录
- 允许用户从祖域页再次打开历史结果
- 允许用户重复使用历史文案或历史模板

### 4.9 缺少完整状态覆盖

当前核心链路可以走通，但状态矩阵还不完整。

需要补齐：

- 首次进入状态
- 本地无数据状态
- 已祭过一次状态
- 有多条历史记录状态
- 海报生成失败状态
- 相册保存失败状态
- 本地缓存被清空后的恢复状态

### 4.10 缺少“本地隐私叙事”

既然本项目强调不上传用户数据，那么产品体验层必须把这点说清楚，而不是只停留在开发口径。

需要补齐：

- 在合适入口明确说明“内容默认只保存在本机”
- 明确“截图、保存、主动分享”才会让内容离开设备
- 提供“清空本地祭祖记录”入口
- 提供“删除单条记录”入口

## P1 本地数据能力必须补齐

### 4.11 缺少统一的本地存储层

当前状态多数还停留在页面内部，没有形成统一的本地缓存封装。

需要补齐：

- 建立统一的 local storage 读写层
- 统一 key 命名规则
- 统一默认值与异常兜底
- 统一序列化 / 反序列化处理

### 4.12 缺少本地数据模型

既然走纯本地路线，就更需要提前把结构定清楚。

建议至少定义：

- 祖先节点数据
- 仪式动作数据
- 用户输入文案数据
- 祭祖记录数据
- 海报记录数据
- 主题和界面偏好数据

### 4.13 缺少本地版本升级策略

后面一旦字段变化，没有版本迁移策略就很容易造成旧数据失效或报错。

需要补齐：

- 本地存储版本号
- 老数据迁移策略
- 无法迁移时的回退策略
- 清空损坏缓存的兜底逻辑

### 4.14 缺少数据管理入口

本地优先产品不应该只负责“写入”，还要负责“让用户管理自己的本地内容”。

需要补齐：

- 清空全部本地祭祖记录
- 删除单条祭祖记录
- 删除本地海报记录
- 重置默认祖先节点或模板

### 4.15 缺少本地导出 / 备份方案

当前不接云端，同步不做，那用户的安全感需要靠“可导出”来补。

需要补齐：

- 是否允许导出本地 JSON
- 是否允许导出海报合集
- 是否提供“手动备份提示”
- 是否在 README 或产品说明里明确“卸载即可能丢失本地数据”

## P2 工程结构仍需收口

### 4.16 API 骨架是否继续保留，需要明确

项目已经明确不接接口，但仓库中仍保留请求层和 health 模块。

当前状态：

- 已明确标注“暂不启用，仅保留工程占位”
- 已在代码层默认禁用远程请求
- 后续如果要重新启用，必须先单独评审

当前不再是“未决策”问题，但后续仍要避免误用。

### 4.17 共享类型和业务结构还不够完整

当前已有基础类型，但还没有围绕本地祭祖记录建立完整共享类型。

需要补齐：

- 本地记录类型
- 祖先节点类型
- 海报记录类型
- 本地缓存 key 类型
- 可复用常量与默认模板类型

### 4.18 主题数据仍然有历史遗留

当前 `theme.ts` 里同时保留了旧方案和赛博方案展示数据。

需要补齐：

- 明确当前 UI 展示页到底展示哪套主题
- 明确旧主题是否仍保留入口
- 避免后续开发错误使用旧 token 描述

### 4.19 缺少本地回归检查清单

既然没有接口，工程质量重点就会落在端内能力稳定性上。

需要补齐：

- 本地写入是否成功
- 页面重进后缓存是否恢复
- 删除记录后首页和祖域是否同步刷新
- 海报生成后是否可再次读取
- 微信端保存相册是否有明确反馈

## 5. 推荐推进顺序

建议按下面顺序推进，而不是并行乱改：

1. 先统一规则文件与文档方向
2. 再补本地优先的数据模型与本地存储层
3. 然后把首页、仪式页、结果页与祖域页接到本地缓存闭环
4. 最后再补导出、清空、隐私说明和发布前内容

---

## 6. 下一阶段最小可执行任务

如果只做一轮最小推进，建议优先完成以下 6 项：

1. 在规则文件中写入“本地优先，不接接口，不上传用户数据”
2. 重写或归档冲突文档，统一产品方向
3. 新建统一本地存储层
4. 定义祭祖记录、祖先节点、海报记录三类核心本地数据结构
5. 让结果页生成后真正写入本地历史
6. 让祖域页开始读取并展示本地历史数据，而不是继续占位

---

## 7. 当前结论

当前项目已经具备：

- 赛博祭祖的视觉方向
- 首页到仪式页到结果页的原型主链路
- 海报生成与保存的基础能力

当前最缺的不是接口，而是：

- 方向统一
- 本地数据闭环
- 祖域第二入口真实化
- 隐私叙事与数据管理能力

一句话总结：

> 下一阶段应该把 `Truth` 做成一个真正“只存在于用户本机、但仍然足够酷、足够离谱、足够有传播感”的赛博祭祖产品。
