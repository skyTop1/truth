# 赛博祭祖本地优先产品缺口清单

版本：v0.1  
更新时间：2026-04-15

## 0. 最新落地进展

截至本轮迭代，以下能力已经落地，不再属于“完全未开始”状态：

- 已补齐统一本地存储层、local id 工具和祭祖归档 store
- 已定义祖先节点、祭祖记录、海报记录等核心本地数据结构
- 首页统计、本机动态、最近祖先节点已经改为读取本地记录
- 仪式页支持自定义祖先称呼、自由输入文案，并可复用本地历史模板
- 结果页生成后会把祭祖记录与海报状态写入本地祖域仓
- 祖域页已经从概念占位页升级为本地管理入口，可查看历史、查看海报、复用历史、删除单条记录、删除单张海报、清空全部记录
- 已增加“复制备份串”能力，用于手动导出当前本地 JSON 快照
- 已补齐本地归档版本迁移与损坏缓存自动恢复策略
- 结果页已补齐海报缓存恢复、缓存失效、生成失败、相册权限受限等关键状态
- 结果页分享路径已改为应用入口，不再携带祖先称呼、悼词等用户输入
- H5 生成的 base64 海报不再写入本地归档，导入备份时会过滤远程图片和不安全海报缓存
- 首页与祖域页已增加首次进入、已有历史但无海报、完整闭环等状态矩阵提示
- 祖域页已支持手动导出与导入恢复，本地备份链路形成闭环
- README、`manifest.json` 与项目结构文档已同步到当前“本地优先 + 赛博祭祖”口径
- 已新增本地优先回归检查清单，静态校验与手工回归入口更明确
- 主题链路已从二元亮暗切换升级到四套运行时主题预设，主题预览页可直接切换

但这不代表清单已经全部完成。

当前仍然需要继续推进的是：

- 更完整的状态矩阵覆盖
- 更明确的本地隐私说明和发布前权限说明
- README 的更完整使用说明与备份说明

补充说明：

- `src/api/*` 当前已明确标记为未启用占位，并在代码层默认禁用远程请求

## 1. 文档目标

本文档用于在当前方向已经明确的前提下，整理本项目下一阶段仍未完善的内容。

这份清单不再讨论“要不要做云端产品”或“要不要接接口”，而是基于当前已确认约束，明确：

1. 哪些内容已经确定，不再反复争论
2. 哪些内容当前不算缺口
3. 哪些内容必须继续补齐
4. 后续应该按什么顺序推进

一句话目标：

> 把 `Truth` 从“能跑的赛博祭祖原型”推进到“方向统一、体验完整、隐私叙事明确、可持续迭代”的本地优先小程序 / H5 产品。

---

## 2. 已确认前提

以下前提已经确认，后续规划默认全部遵守：

- 项目名称固定为 `Truth`
- 产品概念固定为“赛博祭祖”
- `Truth` 作为项目名，与“赛博祭祖”产品概念不冲突
- 当前阶段不接任何接口
- 当前阶段不引入账号体系
- 当前阶段不把用户祭祖内容上传到服务端
- 数据只保存在本地与本地缓存
- 不介入用户祭祖内容，不做服务端审核
- 用户想怎么祭祖就怎么祭祖
- 优先平台仍然是 `微信小程序 + H5`

---

## 3. 当前不算缺口的内容

以下内容不纳入当前阶段缺口：

- 云端接口接入
- 账号登录 / 注册
- 远程用户体系
- 服务端数据存储
- 内容上传到云端
- 远程内容审核
- 远程统计后台
- 多端云同步

这些方向不是“还没做”，而是当前阶段明确“不做”。

## 4. 缺口清单

## P0 必须先统一

### 4.1 规则文件已写清“本地优先”，但后续实现仍需持续遵守

当前 `AGENTS.md` 与 `shrimp-rules.md` 都已经把“禁止引入接口、禁止上传用户祭祖内容、默认本地缓存”写成硬约束。

需要补齐：

- 确保后续页面和功能继续以本地缓存为第一选择
- 确保新增能力不会绕过现有本地优先约束
- 如果未来要引入接口，仍然必须先单独评审

### 4.2 文档方向仍有冲突

当前热点方向已经明确，但仓库里仍保留一份“庄重、长期、数字纪念空间”的旧文档，会持续干扰后续判断。

需要补齐：

- 归档或重写旧版产品概念文档
- 统一所有文档中的产品口径
- 明确当前主线是“热点型、本地优先、赛博祭祖体验”

### 4.3 README 和应用描述已基本同步，但使用说明还可继续补充

当前 `README.md` 和 `manifest.json` 已经切到“本地优先的赛博祭祖原型”口径，不再属于 starter 级占位。

需要补齐：

- README 继续补充更完整的页面说明和回归说明
- README 继续补充主题预设和本地缓存说明
- 保持 `manifest.json` 描述与 README 一致
- 继续明确当前主要目标平台和当前阶段范围

### 4.4 主题叙事主线已形成，但历史文档仍需继续收口

实现层当前已经形成“赛博祭祖 + 四套运行时主题预设”的主线，旧的“黛蓝灰、沉稳正式”文档已转为历史参考，但仓库里仍保留多份历史规划文档。

需要补齐：

- 继续统一“当前主主题”与“历史主题方案”的关系
- 明确哪些文档是历史归档，哪些文档仍然作为当前实现依据
- 避免后续设计继续在“热点赛博”和“正式沉稳”之间来回摇摆

## P1 产品体验必须补齐

### 4.5 祖域第二入口已落地为本地管理入口，但内容层还可继续扩展

当前祖域页已经不再是概念占位，已经具备本地历史、海报、备份导入导出、删除和清空能力。

需要补齐：

- 继续补充常用模板或常用仪式动作入口
- 继续优化长列表下的管理效率和状态提示

### 4.6 首页已接本地数据，但首屏策略还可继续细化

当前首页的统计、最近动态和状态提示已经接到本地记录，不再是纯静态展示页。

需要补齐：

- 继续细化不同数据量级下的首页内容编排
- 继续补充“已有历史但缺海报”等细分状态的引导动作
- 继续校准首页传播区和本地管理区的权重

### 4.7 仪式页自由度已有进展，但模板能力还不够完整

当前仪式页已经支持自定义祖先称呼、自由输入文案和复用最近本地历史模板，不再是完全固定演示流程。

需要补齐：

- 继续扩展仪式动作，而不是长期固定三项
- 继续沉淀本地常用文案模板或常用动作组合
- 继续明确动作上限、模板复用和自由输入之间的产品边界

### 4.8 结果页已形成本地闭环，但传播细节还可继续打磨

当前结果页已经会写入本地祭祖记录、恢复旧记录、恢复或重建海报缓存，也支持从祖域页再次打开历史结果。

需要补齐：

- 继续优化传播态文案和二次生成反馈
- 继续补充更多失败态和恢复态提示
- 继续评估是否要增加更多海报模板或单图样式

### 4.9 缺少完整状态覆盖

当前核心链路可以走通，但状态矩阵还不完整。

需要补齐：

- 首次进入状态
- 本地无数据状态
- 已祭过一次状态
- 有多条历史记录状态
- 海报生成失败状态
- 相册保存失败状态

### 4.10 本地隐私叙事已进入页面，但发布级说明仍不完整

当前首页、仪式页和祖域页都已经把“内容默认只保存在本机”“删除后无法从云端恢复”等口径写进页面文案，删除与清空入口也已经存在。

需要补齐：

- 继续统一产品内、README 和发布材料里的隐私口径
- 继续补充“卸载或清缓存即丢失”的更强提示

## 5. 推荐推进顺序

建议按下面顺序推进，而不是并行乱改：

1. 先继续统一文档主线和历史归档边界
2. 再补本地隐私说明、权限说明和发布前文案
3. 然后继续细化回归检查清单与更多状态矩阵
4. 最后再考虑模板扩展、传播样式扩展和更细的管理能力

## 6. 下一阶段最小可执行任务

如果只做一轮最小推进，建议优先完成以下 6 项：

1. 继续归档冲突文档，明确当前主线文档
2. 把本地优先回归检查清单接入每轮验收
3. 补页面内可见的隐私和权限说明
4. 继续完善主题和本地缓存的使用说明
5. 补更多失败态、空态和恢复态文案
6. 评估下一轮要优先扩模板能力还是管理能力

## 7. 当前结论

当前项目已经具备：

- 四套运行时主题预设
- 首页到仪式页到结果页到祖域页的本地闭环
- 海报生成、恢复、预览、保存和备份导入导出能力

当前最缺的不是接口，而是：

- 更统一的文档主线
- 更完整的隐私叙事与发布说明
- 更系统的回归检查清单
- 更细的状态矩阵和管理体验

一句话总结：

> 下一阶段应该把 `Truth` 做成一个真正“只存在于用户本机、但仍然足够酷、足够离谱、足够有传播感”的赛博祭祖产品。
